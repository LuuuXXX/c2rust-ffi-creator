## 2. 符号对账（⚠ 必须以实测为准）

> **关键原则**：北向接口的最终范围必须由已构建二进制的实测导出符号决定，
> 而不能仅凭头文件声明推断。头文件声明与实际导出符号之间可能因条件编译、
> 宏展开、链接脚本等原因存在差异，所有差异必须在此核对并解释。

### 2.1 符号提取（必须执行，不可跳过）

- [ ] 已实际构建 C 产物（在干净环境中，产物与目标平台匹配）
- [ ] 已运行以下命令之一提取实际导出符号并将结果粘贴至下方：

```bash
# Linux .so / .a
nm --defined-only --extern-only <产物路径> | grep ' T '
# macOS .dylib / .a
nm -U <产物路径> | grep ' T '
# Windows .dll（需 dumpbin）
dumpbin /EXPORTS <产物路径>
```

## 3. 结构体布局 & 类型对齐

- [ ] 所有跨 FFI 传递的结构体均标注 `#[repr(C)]`
- [ ] 已使用 `std::mem::size_of` / `std::mem::offset_of` 断言与 C 侧一致
- [ ] 含位域（bitfield）的结构体已改为显式 `u32 flags` + 位掩码常量，并文档说明

## 6. 内存归属策略

- [ ] 已选定内存归属模式（仅选一个或明确组合）：
  - [ ] **Handle 模式**：`*_new()` 返回不透明指针，`*_free()` 释放
  - [ ] **调用者分配**：调用方分配 buffer，被调用方填充
  - [ ] **混合**：已为每个接口明确标注（见 `spec-v1-extraction.yml`）
- [ ] 跨 FFI 不直接暴露 Rust 分配的可变借用引用
- [ ] 所有 `*_free()` / `*_destroy()` 函数已在 Rust 侧实现并导出

---

## 7. Panic 处理

- [ ] 所有 `extern "C"` 导出函数均包含 `std::panic::catch_unwind`
- [ ] Panic 被转换为具体错误码返回（不使用 `abort`，除非明确决策）
- [ ] 已编写触发 Panic 的单测，验证其被正确捕获并转为错误码

---

## 8. 线程安全声明

- [ ] 对每个导出函数，已在文档中明确声明线程安全性
- [ ] 非线程安全函数已通过内部 Mutex 保护或文档声明要求外部同步
- [ ] 若使用全局状态，已验证其初始化线程安全（`std::sync::Once` 或类似机制）
